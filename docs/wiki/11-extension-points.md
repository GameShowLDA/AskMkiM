# Точки расширения

## Зачем эта страница

В проекте много мест, где новые сценарии подключаются автоматически:

- через reflection;
- через интерфейсы;
- через атрибуты;
- через сервисы загрузки из базы.

Эта страница нужна как короткая карта: что менять, если мы расширяем систему.

## Как добавить новую команду языка

Обычно нужен такой набор шагов:

1. Добавить или уточнить метаданные команды:
   - enum/attribute в `Ask.Core`, если команда относится к общей системе описания.
2. Добавить модель команды в `Ask.Engine/ControlCommandAnalyser/Model`.
3. Добавить parser в `Ask.Engine/ControlCommandAnalyser/Parser`.
4. Добавить formatter в `Ask.Engine/ControlCommandAnalyser/Formatter`.
5. При необходимости добавить builder тела команды в `ComandBody`.
6. Если команда исполняемая, добавить executor в `Ask.Engine/ControlCommandExecutor/Executors`.
7. Обновить подсветку в `UI/MKI_OPKW.xshd`.
8. Добавить примеры, справку и тестовые сценарии.

Почему это работает:

- `CommandTranslationManager` сам находит parsers, formatters и body-builders;
- `CommandExecutorRegistry` сам находит executors.

## Как добавить новый метрологический режим

Обычно нужен такой путь:

1. Добавить `UserControl` режима.
2. Пометить его `MetrologyModeAttribute`.
3. Добавить алгоритм измерения в `Ask.Engine/Tests/Metrology` или соседний пакет логики.
4. Добавить вызов из `MetrologyViewModel`, если режим должен быть доступен в меню.
5. При необходимости добавить локализацию и help-страницу.

Почему это работает:

- `MetrologyControlFactory` регистрирует контролы по атрибуту автоматически.

## Как добавить новый инженерный тест оборудования

Обычно нужно:

1. Создать отдельный алгоритм в `Ask.Engine/Tests`.
2. Добавить UI-контрол в `UI/Controls/ExecutorControls/TestsControls` или соседний раздел.
3. Прописать открытие через `TestService`.
4. При необходимости добавить валидацию ввода и сообщения пользователю.

## Как добавить новое устройство

Минимальный набор обычно такой:

1. Добавить интерфейс или capability в `Ask.Core`, если это новый тип поведения.
2. Добавить concrete device в `NewCore/Device`.
3. Добавить протокол связи в `NewCore/Communication`, если старые не подходят.
4. Добавить менеджеры функций в `NewCore/Function`.
5. При необходимости добавить adapters в `NewCore/FunctionAdapters`.
6. Добавить entity и сервис загрузки/сохранения в `DataBaseConfigruration`.
7. Добавить UI-форму настройки устройства.
8. Добавить миграцию EF, если меняется модель базы.

## Как добавить новую настройку

Обычно нужно пройти всю цепочку:

1. model в `Ask.Core/Shared/Entity/Settings` или соседнем разделе;
2. `DbSet` и миграция;
3. сервис чтения/сохранения в `DataBaseConfigruration/Services/Settings`;
4. инициализация в `DatabaseInitializer`;
5. статический config-manager или расширение существующего;
6. UI-контрол настройки;
7. сохранение обратно через event или service.

## На какие механизмы опирается расширяемость

### Reflection

Используется для:

- parsers;
- formatters;
- builders тела команд;
- executors;
- метрологических контролов.

### Атрибуты

Используются для:

- описания метрологических режимов;
- metadata команд;
- привязки возможностей устройств.

### События

Используются для:

- сохранения настроек;
- UI-реакций;
- управления точками останова;
- runner/translator навигации;
- системных состояний.

## Практический совет

Перед тем как добавлять новый механизм, полезно сначала найти ближайший уже существующий аналог:

- новую команду лучше делать по образцу существующей команды того же класса;
- новый метрологический режим лучше делать по образцу соседнего режима;
- новый тест оборудования лучше делать по образцу `CrossConnectionTests` или одного из method/node executors.

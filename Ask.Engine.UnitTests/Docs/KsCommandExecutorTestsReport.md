# Отчет по тестам `KsCommandExecutor`

Файл с тестами: `Ask.Engine.UnitTests/ControlCommandExecutor/Executors/KsCommandExecutorTests.cs`

Ниже перечислено, что именно проверяет каждый кейс и какой риск он закрывает.

## 1. Ключ `Д` пишет успешное измерение в документирование

- Тест: `ExecuteAsync_WithDocumentationKey_WritesInfoToProtocol`
- Сценарий: обычное измерение сопротивления попадает в допустимый диапазон, у команды есть ключ `Д`.
- Проверяется:
  - ошибок в `ProtocolModel.Errors` нет;
  - в `ProtocolModel.Info` появляется одна запись;
  - используется `ResistanceManager`, а не `ContinuityManager`;
  - выполнено подключение мультиметра и подсветка строки редактора.
- Какой риск закрывает: команда может выполнить измерение, но не задокументировать его в протоколе.

## 2. Ключ `Б` переключает команду в режим прозвонки

- Тест: `ExecuteAsync_WithContinuityKey_UsesContinuityMeasurement`
- Сценарий: у команды есть ключи `Б` и `Д`.
- Проверяется:
  - вызывается `ContinuityManager.SetContinuityModeAsync`;
  - вызывается `ContinuityManager.CheckContinuityAsync`;
  - `ResistanceManager.MeasureResistanceAsync` не вызывается;
  - успешный результат попадает в `Info`.
- Какой риск закрывает: исполнитель может игнорировать ключ `Б` и идти не по той ветке измерения.

## 3. Без ключа `Д` успешное измерение не записывается в документирование

- Тест: `ExecuteAsync_WithoutDocumentationKey_DoesNotWriteInfo`
- Сценарий: измерение успешное, но ключ `Д` не задан.
- Проверяется:
  - ошибок нет;
  - блок `Info` остается пустым.
- Какой риск закрывает: исполнитель может загрязнять протокол лишними измерениями.

## 4. Ошибочное измерение формирует ошибку протокола и execution error

- Тест: `ExecuteAsync_WhenMeasurementFails_WritesErrorAndPublishesExecutionError`
- Сценарий: измеренное значение выходит за допустимый диапазон.
- Проверяется:
  - в `Errors` появляется одна запись;
  - в `Info` ничего не добавляется;
  - наружу публикуется `ErrorItem`;
  - в описании ошибки присутствуют точки `X1` и `X2`.
- Какой риск закрывает: команда может молча проваливаться без фиксации брака.

## 5. Ошибка при наличии ключа `Д` попадает и в ошибки, и в документирование

- Тест: `ExecuteAsync_WhenMeasurementFailsWithDocumentationKey_WritesErrorAndInfo`
- Сценарий: измерение не прошло, но ключ `Д` включен.
- Проверяется:
  - в `Errors` есть запись;
  - в `Info` тоже есть запись с фактическим измерением;
  - публикуется `ErrorItem`.
- Какой риск закрывает: при браке может теряться фактическое измеренное значение, нужное для анализа.

## 6. При открытой верхней границе используется `firstValue + 10`

- Тест: `ExecuteAsync_WithoutHigherLimit_UsesLowerLimitPlusTenAsMeasurementTarget`
- Сценарий: у команды задан только нижний предел.
- Проверяется:
  - в вызов измерения уходит целевое значение `firstValue + 10`;
  - верхняя граница передается как `-1`;
  - успешное измерение документируется.
- Какой риск закрывает: исполнитель может использовать неверную целевую точку измерения.

## 7. В рабочем режиме вычитается сопротивление коммутатора

- Тест: `ExecuteAsync_InWorkingMode_SubtractsSwitchResistanceBeforeValidation`
- Сценарий: прибор возвращает значение на границе, но у релейного модуля есть сопротивление переключения.
- Проверяется:
  - после вычитания `SwitchResistance` результат выходит из диапазона;
  - команда пишет ошибку.
- Какой риск закрывает: можно получить ложную норму, если не учитывать паразитное сопротивление коммутатора.

## 8. Отрицательное значение после вычитания прижимается к нулю

- Тест: `ExecuteAsync_WhenSwitchResistanceMakesValueNegative_ClampsMeasurementToZero`
- Сценарий: после коррекции на `SwitchResistance` измерение уходит ниже нуля.
- Проверяется:
  - в сообщение об ошибке попадает `0`, а не отрицательное число;
  - публикуется `ErrorItem`.
- Какой риск закрывает: пользователь может увидеть физически бессмысленное отрицательное сопротивление.

## 9. В idle-режиме сопротивление коммутатора не вычитается

- Тест: `ExecuteAsync_InIdleMode_DoesNotSubtractSwitchResistance`
- Сценарий: включен `IdleMode`, симуляция ошибок выключена.
- Проверяется:
  - то же измерение, которое в рабочем режиме дало бы ошибку, здесь проходит успешно;
  - ошибок и документирования нет.
- Какой риск закрывает: логика idle-режима может случайно совпасть с рабочей и искажать учебный/проверочный прогон.

## 10. В idle-режиме с симуляцией ошибок формируется принудительный брак

- Тест: `ExecuteAsync_InIdleModeWithErrorSimulationAndOpenUpperLimit_ForcesFailure`
- Сценарий: включены `IdleMode` и `IsErrorSimulationMode`, верхняя граница диапазона открыта.
- Проверяется:
  - даже если прибор вернул допустимое значение, в итоговом результате формируется ошибка;
  - публикуется `ErrorItem`.
- Какой риск закрывает: режим имитации ошибок может не сработать и не дать проверить реакцию системы на брак.

## Что не покрыто этим набором

- разбор текста команды `КС` на уровне парсера;
- ошибки трансляции `KS001-KS005`;
- поведение на более сложных схемах с несколькими фрагментами и рекурсивным разбиением цепи;
- интеграция с реальными приборами и фактическими WPF-экранами.
